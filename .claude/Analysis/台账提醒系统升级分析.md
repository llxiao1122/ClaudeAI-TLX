# 台账提醒系统升级分析

> **时间：** 2026-07-07 → 2026-07-08
> **类型：** 系统能力迭代
> **触发：** 用户指出 `工班长每日核查台账清单.xls` 需要整合到每日工作提醒

---

## 一、背景

铁炉西工班有 20 项安全台账，涉及消防、设备、物资、培训、演练、隐患排查等，核查周期各异（每日/每周五/每月固定日/季度末）。此前这些台账信息仅存储为 `.xls` 静态文件，与每日工作提醒（`today.md`）分离，容易被遗漏。

用户提出"每天工作提醒应包含这些台账的提醒"——要求将台账核查整合到日常 AI 工作秘书系统中。

## 二、实施内容

### 2.1 做了什么

| 步骤 | 产出 | 说明 |
|:----|:-----|:-----|
| ① 解析 XLS | 识别20项台账及周期 | 使用 xlrd 读取 `.xls` 并归档为结构化数据 |
| ② 建立参考文档 | `Knowledge/.../台账清单-提醒参考.md` | 按周期分类，便于人查阅 |
| ③ 编写自动脚本 | `Commands/台账提醒/gen_daily_check.py` | 根据当天日期，自动计算需检查项（支持指定日期） |
| ④ 集成到上下文 | `today.md` 新增「今日台账核查」章节 | 每日工作上下文直接包含台账提醒 |
| ⑤ 注册为技能 | 系统自动注册 `/台账提醒` | 可随时调用 |
| ⑥ 建立长期记忆 | `Memory/AccountsLedgerCheck.md` | 跨会话保持一致性 |
| ⑦ 动态维护 | 起重机/叉车移交→即时更新脚本 | 数据变更后 5 分钟内完成全链路更新 |

### 2.2 系统能力提升

**之前：**
- 台账清单是静态 `.xls` 文件
- 每天需手动对照 20 项判断需要查哪些
- `today.md` 仅有一条"每日台账进度16:00前"的笼统提醒
- 移交/变更后文件无人更新

**之后：**
- 脚本自动判断当天需查项（仅2项日常，周五触发5项，18日触发3项等）
- 节假日自动提醒提前
- 数据变更可快速同步（脚本+参考文档+上下文+记忆四层联动）
- `/台账提醒` 技能随时可调

## 三、关键思考

### 3.1 不要一次性做大而全

第一次实现时我打算做一个很完整的系统，但实际上用户的需求很具体：**每天工作提醒包含这些**。所以：
- 先出最小可用版本（Python 脚本 + today.md 集成）
- 再根据使用反馈迭代（用户提到叉车移交，即时更新）

### 3.2 三层数据模型值得保留

这次形成了清晰的三层结构：

```
Knowledge/         ← 原始数据 + 参考（适合人阅读）
Commands/脚本      ← 自动化处理（适合机器执行）
Memory/            ← 跨会话上下文（适合AI记忆）
today.md           ← 每日执行上下文（当天行动）
```

这种分层的好处：修改底层数据（台账变动）时，只要更新 Knowledge 和脚本，Memory 和 today.md 自动联动调整。

### 3.3 源文件依然是权威来源

XLS 是原始清单，提醒参考 Markdown 和脚本都是派生品。下次台账正式变动时，仍然以 XLS 为准，再同步更新派生文件。但这次实践表明，**脚本+参考文档的更新速度可以很快**（5分钟内完成全链路），所以源文件和派生文件之间的同步不是什么大问题。

### 3.4 可改进的方向

1. **负责人列缺失** — 目前表格没有明确的"责任人"，后续可以补上，脚本也能据此分配提醒到具体人
2. **季度提醒不够智能** — 季度末最后一个工作日的计算是硬编码，可以用更通用的算法
3. **台账变动时的联动通知** — 如果有台账新增/删除，目前需要手动更新脚本和参考文档，可以设计一个更自动化的联动机制

## 四、总结

这次升级完成了**台账提醒从静态文件到动态系统**的转变。核心收益不是自动化本身，而是：
- **减少认知负担** — 不用每天想"今天该查什么"
- **减少遗漏风险** — 系统自动提醒，不会因为忙而漏掉
- **快速响应变化** — 移交/新增/删除都能在几分钟内同步

这是一个典型的"小投入、大回报"的系统改进：2 小时不到完成初始搭建，之后每次台账变动只需 <5 分钟维护。
