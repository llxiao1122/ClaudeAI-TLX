# 企业大脑（Enterprise_Brain）搭建总结

> 搭建时间：2026-07-08
> 目的：将 Knowledge/ 原始资料库提炼为 AI 可调用的企业认知模型
> 触发条件：用户询问"企业大脑"建设过程 / 需要回顾认知层建设逻辑时读取

---

## 一、搭建目的

将 Knowledge/ 中的 168 个原始文件（doc/docx/xlsx/pptx/pdf）中的管理逻辑提炼出来，建立 AI 可快速调用的**认知模型**。解决三个核心问题：

| 问题 | 方案 |
|:-----|:------|
| 原始文件不可直接读取（二进制格式） | 提炼为结构化 Markdown 认知层 |
| 制度分散在 11 个目录中，关联不明 | 建立跨目录关联索引 |
| 版本新旧不分、冲突不明 | 建立制度全生命周期管理 |

## 二、最终结构

```
Enterprise_Brain/         ← AI理解后的企业认知层
├── Enterprise_Brain.md   ← 业务关键词→知识文件 速查索引
├── 01_Organization/      ← 组织人事：谁是谁、谁负责什么、谁审批什么
├── 02_Leadership/        ← 领导决策模型：关注重点、汇报策略、风险敏感点
├── 03_Management/        ← 制度版本管理：全生命周期追踪
├── 04_Evolution/         ← 增量学习：变更记录 + 学习SOP
├── Core/                 ← 企业全景 + 使用指引
├── Management/            ← 审批逻辑 + 风险识别
└── Operations/            ← 供应链/安全/台账/版本/索引
```

**Knowledge/** = 原始资料库（不可直接读取）
**Enterprise_Brain/** = AI 理解后的认知层（可直接查询）

## 三、搭建步骤

### 步骤1：全面扫描

```
find Knowledge/ -type f | sort  → 168文件, 11模块
读取每个子目录的 README.md     → 理解模块管理目的
读取文件名分析版本号规则         → YYYY-名称-版本号.ext
```

### 步骤2：认知层提取

对每个模块提取：
- 管理目的（为什么要有这个制度）
- 核心规则（流程逻辑/控制点）
- 权责关系（谁审批/谁执行/谁监督）
- 风险控制点（容易出什么问题）
- 例外路径（不走常规流程的情况）

### 步骤3：组织人事建模

从 `.claude/Management/People/`（6人完整档案）+ `Leaders/`（8位领导画像）+
`Rules/organization.md`（分工规则）+ 事件记录中提取：
- 岗位职责树（谁负责什么台账）
- 审批权矩阵（谁审批什么事项）
- 汇报链路（有事找谁、怎么汇报）

### 步骤4：领导决策建模

从实际工作记录中提取（非性格推测）：
- 10位领导的关注重点和检查方式
- 4种工作部署模式
- 风险敏感点和触发信号
- 汇报偏好和沟通策略

### 步骤5：制度版本追踪

建立全库版本生命周期：
- 版本命名规则解析
- 替代关系链（如 2022-危废→2023-修订）
- 冲突标记（6处待确认项）
- AI引用优先级策略

### 步骤6：关键词索引

建立业务关键词→认知层/制度层 双向映射：
- 13个分类，约60个关键词
- 每个词指向 Enterprise_Brain/ 认知文件 和 Knowledge/ 制度文件

## 四、设计原则

| 原则 | 说明 |
|:-----|:------|
| 不复制原文 | 只提炼管理逻辑，不照搬制度条文 |
| 不确定不推断 | 无法确认版本号/内容差异的，标记"需人工确认" |
| 来源可追溯 | 每个知识点标注来源文件 |
| 极简 | 以最小 token 量包含最大信息密度 |
| 三重分离 | 制度=约束、手册=操作、台账=证据 |
| 只基于证据 | 领导画像仅来自可验证的工作记录 |

## 五、使用方式

```
需要"制度" → Enterprise_Brain.md 关键词索引 → 定位认知文件 → 调 Knowledge/ 原件
需要"操作" → Operations/05_知识关联索引 → 定位手册 → 调 03-工作指导手册/
需要"人"   → 01_Organization/ → 组织架构/岗位职责/权责/汇报
需要"领导" → 02_Leadership/领导决策模型
需要"版本" → 03_Management/制度版本管理
```
